## Phase 4: Ingestion Service

All files in `~/dev/android-re-pipeline/services/ingestion/`.

### Task 4.1 — APKSource interface

**Create** `internal/sources/source.go`:

```go
package sources

import (
    "context"
    "io"
)

type APKMetadata struct {
    PackageName string
    Version     string
    VersionCode int
    Source      string
    DownloadURL string
    SHA256      string // hex-encoded, computed after download
}

type APKSource interface {
    Name() string
    Download(ctx context.Context, identifier string) (io.ReadCloser, *APKMetadata, error)
}
```

The `identifier` parameter is a URL for DirectURL, or a package name for APKPure.

### Task 4.2 — DirectURL source

**Create** `internal/sources/directurl.go`:

- Implements `APKSource`
- `Name()` returns `"directurl"`
- `Download()`:
  - HTTP GET the provided URL with a reasonable timeout (5 minutes)
  - Stream the response body back as `io.ReadCloser`
  - Set `DownloadURL` in metadata
  - PackageName can be left empty (will be populated later from APK parsing)
  - SHA256 is computed by the caller after download completes (see ingestion handler)

> **Parking lot:** APKPure and other scraping-based sources deferred to future
> work. The `APKSource` interface is designed to make adding new sources easy.

### Task 4.4 — NATS publisher

**Create** `internal/queue/publisher.go`:

- Struct `Publisher` wrapping `*nats.Conn` and `nats.JetStreamContext`
- `NewPublisher(natsURL string) (*Publisher, error)`:
  - Connect to NATS
  - Get JetStream context
  - Create stream `APK_EVENTS` with subjects `["apk.>"]` if not exists
    - MaxDeliver: 5 (retry limit)
    - Retention: WorkQueuePolicy
- `PublishIngested(ctx context.Context, msg *IngestedMessage) error`:
  - Marshal to JSON
  - Publish to `apk.ingested`
- `PublishFailed(ctx context.Context, msg *FailedMessage) error`:
  - Publish to `apk.ingestion.failed`

Message types:
```go
type IngestedMessage struct {
    JobID       string `json:"job_id"`
    APKPath     string `json:"apk_path"`
    PackageName string `json:"package_name"`
    Version     string `json:"version"`
    Source      string `json:"source"`
    SHA256      string `json:"sha256"`
    SubmittedAt string `json:"submitted_at"` // RFC3339
}

type FailedMessage struct {
    JobID string `json:"job_id"`
    Error string `json:"error"`
}
```

### Task 4.5 — HTTP handlers

**Create** `internal/api/handlers.go`:

Dependencies: publisher, data path (from env or flag)

**`POST /upload`**:
1. Parse multipart form, read file from `file` field
2. Generate UUID job_id
3. Write APK to `/data/apks/<job_id>.apk`, computing SHA256 as it writes (use `io.TeeReader` with `crypto/sha256`)
4. Publish `IngestedMessage` to NATS
5. Return `201 {"job_id": "..."}`
6. On failure at any step, return appropriate 4xx/5xx with `{"error": "..."}`

**`POST /download`**:
JSON body:
```json
{
  "source": "directurl",
  "identifier": "https://..."
}
```

1. Parse and validate body
2. Generate UUID job_id
3. Select APKSource by `source` field
4. Call `source.Download(ctx, identifier)` — **synchronous, blocks until download completes**
5. Write streamed APK to `/data/apks/<job_id>.apk`, computing SHA256
6. Publish `IngestedMessage` to NATS
7. Return `201 {"job_id": "..."}`
8. On failure: return 4xx/5xx with `{"error": "..."}` — do NOT publish to NATS on sync failure, just return the error to the caller
9. Set a generous request timeout (10 minutes) for large APK downloads

**`GET /health`**:
Return `200 {"status": "ok"}`

### Task 4.6 — Main entrypoint

**Create** `cmd/server/main.go`:

1. Read config from env: `NATS_URL`, `SERVICE_PORT` (default 8080), `DATA_PATH` (default `/data`)
2. Create NATS publisher
3. Create sources (directurl only for now)
4. Set up `net/http` ServeMux with handlers (Go 1.26 supports method+path routing: `mux.HandleFunc("POST /upload", ...)`)
5. Ensure `/data/apks/` directory exists (create if not)
6. Start HTTP server with graceful shutdown (listen for SIGTERM/SIGINT)

  - `re-apktool-<jobID>` with args `["--apk", apkPath, "--output", "/data/output/<jobID>/apktool"]`
- Common spec for both:
  - Labels: `re-tools/job-type: analysis`, `re-tools/job-id: <jobID>`, `re-tools/tool: jadx|apktool`
  - Volume mount: `re-tools-data` PVC at `/data`
  - `restartPolicy: Never`
  - `backoffLimit: 1`
  - `ttlSecondsAfterFinished: 3600`
  - `activeDeadlineSeconds: 600` (10 min timeout)
  - SecurityContext: runAsNonRoot, runAsUser 1000, no privilege escalation, drop all caps
  - Resources: jadx gets 2Gi/1CPU limits; apktool gets 1Gi/500m limits
  - imagePullSecrets: `ghcr-pull-secret`

**`WatchJobs(ctx, handler JobEventHandler)`**:
- Use a SharedInformer on Jobs in `re-tools` namespace filtered by label `re-tools/job-type=analysis`
- On Job completion (succeeded condition): call `handler.OnJobComplete(jobID, tool)`
- On Job failure: capture pod logs via `clientset.CoreV1().Pods().GetLogs()`, call `handler.OnJobFailed(jobID, tool, logs)`

**`ReconcileRunningJobs(ctx, runningJobs []uuid.UUID) error`**:
- For each running job_id from the DB, check if corresponding k8s Jobs exist and their status
- If Job completed but DB says running: trigger completion handler
- If Job doesn't exist: mark as failed with "job not found after coordinator restart"

### Task 5.6 — NATS consumer

**Create** `internal/queue/consumer.go`:

- Connect to NATS JetStream
- Create/ensure stream `APK_EVENTS` on subjects `["apk.>"]`
  - MaxDeliver: 5
  - Retention: WorkQueuePolicy
- Create durable consumer `coordinator` on `apk.ingested`
- On message:
  1. Deserialize `IngestedMessage`
  2. **Dedup check**: query DB by SHA256 — if a completed job exists for this SHA256, log and ack without re-processing
  3. Insert job into DB (status: pending)
  4. Create k8s Jobs for jadx + apktool
  5. Update job status to `running`
  6. Ack message
  7. On error: Nak for redelivery (NATS will enforce MaxDeliver=5)

Also subscribe to `apk.ingestion.failed`:
- Update job status to `failed` with the error message

### Task 5.7 — Pipeline orchestration

This is the core logic that ties job watching to subsequent steps. Can live in `internal/pipeline/orchestrator.go` or be part of the consumer/main package.

**`OnToolJobComplete(jobID uuid.UUID, tool string)`**:
1. Update tool status in DB (e.g., jadx_status = "complete")
2. If tool == "apktool": run metadata parser, insert APK metadata into DB
3. Check if BOTH jadx and apktool are complete:
   - If yes: kick off MobSF submission (inline, not a Job)
     - Upload APK to MobSF
     - Poll for scan completion
     - Store report JSON in DB
     - Update mobsf_status to "complete"
     - Update overall job status to "complete", set results_path to `/data/output/<jobID>`
   - If no: do nothing, wait for the other tool

**`OnToolJobFailed(jobID uuid.UUID, tool string, logs string)`**:
1. Update tool status to "failed"
2. Update overall job status to "failed" with error = tool + ": " + logs
3. Note: if one tool fails, we could still let the other complete. For v1, fail the whole job.

**`GET /status/{job_id}`**:

**`GET /results/{job_id}`**:
- Query job — must be status "complete"
- Return structured results:
  ```json
  {
    "job_id": "...",
    "status": "complete",
    "results_path": "/data/output/<job_id>",
    "package_name": "...",
    "jadx": { "status": "complete", "path": "/data/output/<job_id>/jadx" },

  }
  ```

**`GET /results/{job_id}/tree?path=<relative_path>`**:
- Default path: "" (root of job output)
- List files and directories at the given path under `/data/output/<job_id>/`
- **Path traversal protection**: resolve the full path and ensure it starts with `/data/output/<job_id>/`
- Return:
  ```json
  {
    "path": "jadx/sources/com/example",
    "entries": [
      { "name": "MainActivity.java", "type": "file", "size": 12345 },
      { "name": "utils", "type": "dir" }
    ]
  }
  ```
- Limit to 1000 entries per response

**`GET /results/{job_id}/file?path=<relative_path>`**:
- Path traversal protection (same as tree)
- Read file from `/data/output/<job_id>/<path>`
- Return raw text content
- Truncate at 100KB with `X-Truncated: true` header
- 404 if file doesn't exist

**`GET /results/{job_id}/search?q=<query>&max=<limit>`**:
- Path traversal protection on base dir
- Run `grep -r --include='*.java' --include='*.smali' --include='*.xml' --include='*.json' -l` (or implement in Go with `filepath.Walk` + `strings.Contains` for simplicity)
- Default max results: 50, configurable up to 200
- Timeout: 30 seconds
- Return:
  ```json
  {
    "query": "...",
    "matches": [
      { "file": "jadx/sources/com/example/Api.java", "line": 42, "context": "private static final String API_URL = \"https://...\";" }
    ],
    "truncated": false
  }
  ```

**`GET /health`**:
Return `200 {"status": "ok"}`

## Phase 7: Claude Skill

### Task 7.1 — Skill definition

**Create** `skill/SKILL.md`:

The skill should instruct Claude on:

1. **Base URLs** (configurable, default to ingress URLs):
   - Ingestion: `https://ingestion.apps.blj.wtf`
   - Coordinator: `https://coordinator.apps.blj.wtf`

2. **Submitting APKs**:
   - Upload: `curl -X POST -F 'file=@app.apk' <ingestion>/upload`
   - Download from URL: `curl -X POST -H 'Content-Type: application/json' -d '{"source":"directurl","identifier":"<url>"}' <ingestion>/download`

3. **Checking status**:
   - `GET <coordinator>/status/<job_id>`
   - Poll every 15s until status is `complete` or `failed`
   - Typical analysis takes 2-5 minutes

4. **Browsing results**:
   - List files: `GET <coordinator>/results/<job_id>/tree?path=jadx/sources`
   - Read file: `GET <coordinator>/results/<job_id>/file?path=jadx/sources/com/example/MainActivity.java`
   - Search: `GET <coordinator>/results/<job_id>/search?q=https://&max=50`

5. **Common analysis workflows**:
   - **Find API endpoints**: search for `http`, `https`, `retrofit`, `okhttp`, `volley`
   - **Find hardcoded secrets**: search for `api_key`, `secret`, `token`, `password`, `Bearer`
   - **Review permissions**: `GET .../file?path=apktool/AndroidManifest.xml`
   - **Examine app structure**: browse `jadx/sources/` tree, then read key files
   - **Check for obfuscation**: look at class/method names in jadx output — short single-letter names indicate ProGuard/R8
   - **Find native libraries**: browse `apktool/lib/` tree
   - **Review MobSF findings**: check `mobsf` section in results response for security score and findings

6. **Tips**:
   - jadx output is decompiled Java — easier to read but may have decompilation artifacts
   - apktool output is smali (Dalvik bytecode disassembly) + raw resources — more accurate but harder to read
   - Start with jadx for readability, fall back to smali when jadx output is incomplete
   - Large apps may have thousands of files — use search before browsing

**Acceptance:** Skill is clear, actionable, and covers the full workflow from submission to analysis.

## Parking Lot (deferred)

- **APKPure source** — scraping-based, fragile, ToS concerns. Add later once DirectURL flow is proven.
- **APKMirror source** — more sophisticated scraping protections, even harder than APKPure.
- **Google Play source** — requires account token management, high operational overhead.
- **Archive.org source** — good for truly dead apps, straightforward HTTP, no ToS concerns. Low-hanging fruit for a second source.
- **Version diffing** — jadx output + `diff` or AST-level diff for comparing APK versions.
- **gVisor sandboxing** — stronger isolation for Job pods if untrusted APK concern grows.
- **Pull-through registry cache** — if Job cold-start time becomes noticeable.
- **Result retention policy** — CronJob to clean up old output from PVC after N days.
- **Ingress authentication** — Authelia or API key auth on ingestion/coordinator endpoints.
- **MobSF shared Postgres compatibility** — verify MobSF works with non-superuser role. May need standalone Postgres if it doesn't.
